## 1. Independent Agent Findings

Two background agents were run in parallel to provide independent analysis of (a) dataset realism and (b) generator logic.

### Agent A — Dataset Realism Analysis

**Findings:**
- The existing `data/processed/toy_*.csv` files (n=20, generated by the v3 toy generator) produced **27% machine utilization** in the simulation — far below the 75–85% target stated in the generator's config.
- Root cause: The v3 generator (`generate_toy_company_csv.py`) included an outlier operation `Fre_52` with a duration of ~6,451 minutes. This single outlier inflated `total_work_est`, which inflated `arrival_span`, which spread job arrivals too far apart. With sparse arrivals, machines are almost always idle.
- The `realized_util` parameter was sampled from `U(0.75, 0.95)` **before** computing the arrival span, but the sampled value was used in the denominator — a circular dependency. If `realized_util=0.90`, the arrival span shrinks, which actually *reduces* utilization because fewer jobs overlap.
- All DQN experiments (vanilla, double, dueling, D3QN) ran on this stale dataset: 100% of decision steps had only 0 or 1 dispatchable job. The action mask forced deterministic behavior. All 4 variants produced bitwise-identical episode statistics (`reward=-44610.35`, `util=0.2735`, `makespan=45560.0` every episode).
- All 8 heuristics (EDD, SPT, FIFO, LIFO, LPT, LDD, SLACK, CR) also produced identical episode stats for the same reason — `min(dispatchable, key=fn)` with a single-element list returns that element regardless of `fn`.

**Verdict**: The dataset must be regenerated. The v3 generator has a utilization calibration bug and an outlier duration that makes it unusable for RL training without intervention.

### Agent B — Generator Logic Analysis

**Findings (generate_dataset_final.py, the OLD generator):**

Strengths:
- Clean `@dataclass Config` with CLI overrides via `argparse`.
- Per-operation-type lognormal duration sampling with explicit `mu`, `sigma` parameters.
- Arrival time model: uniform over a fixed window `ARRIVAL_MAX = 7 * 24 * 60 = 10,080 min`.
- Validation block checks for empty eligibility lists, zero-duration ops, missing machine names.

Weaknesses identified:
1. **Wrong output format**: Writes `jobs.csv`, `job_ops.csv`, `op2machines.csv` — none of which match the 3-file schema (`{prefix}_jobs.csv`, `{prefix}_operations.csv`, `{prefix}_eligibility.csv`) that `preprocess_company_real.load_company_dataset_from_csv()` expects. Dataset was never loadable by the current env.
2. **Fixed single route for all jobs**: Every job follows the same 8-operation sequence (`Fre_20, Fre_30, Fre_40, Fre_50, Fre_51, Fre_60, Fre_70, Fre_22`). With all jobs identical in structure, SPT/LPT/EDD/etc. degenerate to tie-breaking.
3. **Arrival window too wide**: 10,080-minute window for 20 jobs with ~2,000 min total work → extremely sparse, ~5% utilization.
4. **Duration clip too aggressive**: CLI default `--duration-clip-max 120` caps all durations at 120 min. Real shop operations (Fre_40, Fre_43) need 500–800 min. Clipping destroys variance and realism.
5. **Only 5 machines**: `F2_1, F2_2, F2_3, F1_2, F1_3` — missing `F2_4, F2_5`. The env's score=2 filter further reduces eligible machines per operation, increasing underloading.

**Findings (generate_toy_company_csv.py, the NEW v3 generator):**

Strengths:
- Utilization-aware `arrival_span` formula: `ceil(N * avg_work / (M * target_util))`.
- 6 route templates with Jaccard novelty filter (prevents identical jobs).
- DDT/RDD deadline model.
- Compound Poisson batch arrivals.
- Correct 3-file output schema.

Weaknesses:
1. **Excel dependency**: Requires real company Excel file via `preprocess_company_real.preprocess_company_data()`. Blocks synthetic standalone use.
2. **Fre_52 outlier**: Duration ~6,451 min inflates `avg_work` estimation → `arrival_span` becomes ~8× too large → chronic underloading.
3. **Target_util misapplied**: Used as denominator in arrival_span formula but never calibrated against simulation. Simulation utilization is ~27% vs 75% target.
4. **Over-engineered**: 678 lines for what should be a 200-line self-contained generator. Hard to audit, hard to debug.

## 2. Consolidated Diagnosis

### Primary Bug: Underloaded Dataset → Zero Action Variance

The causal chain:
```
Fre_52 outlier (6451 min)
→ inflates total_work_est
→ inflates arrival_span (jobs arrive over 30,000+ min)
→ at any simulation timestep, 0 or 1 job has arrived AND has an idle eligible machine
→ env._get_dispatchable_jobs() returns 0 or 1 job every step
→ action_mask forces no-op OR the single job
→ all policies (RL and heuristics) make the identical sequence of choices
→ all episode statistics are bitwise identical
→ DQN reward signal = 0 gradient → flat learning curves
```

This explains **both** failure modes observed in experiments:
- Flat DQN learning curves (4 variants × hundreds of episodes = identical)
- Identical heuristic outcomes (8 rules × all episodes = identical)

### Secondary Issue: Stale CSV Files

The `data/processed/toy_*.csv` files were never regenerated after the v2 → v3 generator transition. The files in the repository correspond to a partially-fixed v2 output that still has the above underloading bug. Re-running `generate_toy_company_csv.py` with the current code would regenerate correctly — but the Excel file dependency blocks this.

### The Action Masking Is Correct

`env_company.py`'s action masking logic is **not** the bug. The mask allows no-op only when `len(dispatchable) == 0`, and correctly exposes all dispatchable jobs. The environment logic is sound. The dataset is the problem.

## 3. Code Edits Made

### Created: `generate_toy_synthetic.py`

A fully self-contained generator, based on `generate_dataset_final.py`'s clean structure, with the following improvements:

| Dimension | Old Generator | New Generator |
|-----------|---------------|---------------|
| Output format | Wrong 4-file schema | Correct 3-file schema for `load_company_dataset_from_csv()` |
| Machines | 5 (`F2_1–3, F1_2–3`) | 7 (`F2_1–5, F1_2–3`) — matches env |
| Route variety | 1 fixed 8-op route | 6 distinct templates (5–6 ops each) |
| Arrival model | Uniform over 10,080 min | Poisson with utilization-calibrated span |
| Duration clip | 120 min max | 800 min max (matches real data range) |
| Duration baseline | Short (~100–300 min μ) | Realistic (~80–700 min μ per op-type) |
| Excel dependency | None | None (standalone) |
| Due-date model | Uniform window | DDT ratio: U(1.3, 1.8) × total_job_work |

**Key design choices:**
- `arrival_span = ceil(N * avg_work / (M * target_util))` — same as v3 but without the Fre_52 outlier inflating avg_work.
- Lognormal durations per op-type with medians matching real company data (`Fre_40`: 700 min, `Fre_43`: 650 min, etc.).
- Eligibility scores hardcoded from the company's real machine–operation mapping (score=2: primary capability, score=1: secondary/backup).
- Job-to-template assignment by round-robin (reproducible, balanced route distribution).

**No other files were modified.** The generator is self-contained and writes only to `data/processed/`.

## 6. Realism and Company-Fit Assessment

### Strengths of New Generator

1. **Operation-type durations match real data**: `Fre_40` (large machining): median 700 min; `Fre_43` (assembly): median 650 min; `Fre_22` (finishing): median 80 min. These align with the company's real job card data.
2. **Machine partition preserved**: Score=2 filter keeps the three functional groups (A: F2_1–F2_3 rough machining; B: F2_4–F2_5 precision; C: F1_2–F1_3 assembly). Cross-group eligibility via score=1 provides realistic backup routing.
3. **Route diversity**: 6 templates reflect different job families (small casting, large casting, precision parts, assemblies, etc.). Round-robin assignment ensures a realistic mix.
4. **Dynamic arrivals**: Poisson arrival model with utilization targeting produces a realistic inter-arrival pattern. Jobs don't arrive uniformly — there are bursts and gaps.
5. **Due-date realism**: Slack ratio U(1.3, 1.8) gives 30–80% slack on total job work. Tight enough to force competition; loose enough for feasible schedules.

### Remaining Realism Gaps

1. **No sequence-dependent setup times**: Real shop has machine-specific setup times between job families. Current env doesn't model this.
2. **No machine breakdowns**: Real shop has stochastic downtime events (MTTR, MTBF). These would add urgency to deadline management.
3. **No job priorities**: Real system has rush orders (priority=1) vs. standard (priority=2). The generator could add a priority field.
4. **No partial eligibility**: All ops on a route must have at least one score=2 machine. In reality some ops have only score=1 machines (secondary capability). This could be relaxed.
5. **Simulation utilization gap**: Theoretical `realized_util`=0.74–0.77 vs. simulation utilization 0.49–0.56. The gap is real and expected (scheduling contention, discretization) but the gap size suggests the arrival span formula may need a correction factor of ~0.7 to hit simulation utilization targets.

### Company-Fit Score: 7/10

The generator produces instances that look like a plausible subset of the company's job mix. The major gaps (no setup times, no breakdowns) are deliberate simplifications appropriate for a first-pass RL training dataset.

## 7. Remaining Issues and Next Actions

### Critical (Must Fix Before RL Training)

1. **Decide on n_jobs for training**:
   - n=20 (mean_dispatchable=1.34, 23.6% multi-choice): Easy. Good for debugging reward shaping and verifying the training loop.
   - n=30 (mean_dispatchable=2.83, 47.9% multi-choice): Harder. Better for RL to learn non-trivial policies.
   - **Recommendation**: Use n=30, target_util=0.85 as the primary training configuration. Keep n=20 as a "sanity check" environment.

2. **Regenerate the canonical toy_*.csv files**:
   ```bash
   python generate_toy_synthetic.py --n-jobs 30 --seed 42 --target-util 0.85 --prefix toy
   ```
   This overwrites `data/processed/toy_*.csv` with a loadable, correctly-loaded dataset.

3. **Fix the env seed bug**: `env_company.py` reset() receives a seed but `_rng` is never used by `_reset_state()`. All resets return the same deterministic state. Fix:
   - Either: randomize job arrival times (±jitter) based on the seed.
   - Or: load from a pre-generated pool of datasets (load dataset index = f(seed)).
   Without this fix, DQN still trains on 100% identical episodes.

4. **Retrain DQN after dataset fix**: The flat learning curves were entirely caused by the underloaded dataset + deterministic reset. After fixing both, training should show meaningful learning.

### High Priority (Next Week)

5. **Arrival-span correction factor**: Simulation utilization is ~0.67× of `realized_util`. Adjust the formula:
   ```python
   arrival_span = ceil(N * avg_work / (M * target_util * 0.67))
   ```
   Or: run a short simulation warmup in the generator to calibrate empirically.

6. **Multi-seed training**: Once the seed bug is fixed, train with `episode_seed = base + ep_id` and verify that different seeds produce genuinely different episode statistics (different makespans, different rewards).

7. **Benchmark against heuristics**: Once RL shows a non-flat learning curve, compare its learned policy against EDD (best on tardiness) and SLACK (best on makespan). A well-trained RL policy should achieve makespan ≤ SLACK and tardiness ≤ EDD simultaneously — which no single heuristic can do.

### Medium Priority

8. **Generate benchmark bank**: `data/benchmark/` artifacts use the old schema (4-file format). These need to be regenerated with the new generator if used for evaluation.

9. **PPO vs DQN**: PPO with MaskablePPO (`sb3_contrib`) was the original plan. Now that the dataset is fixed, DQN and PPO should be compared on the same environment. PPO typically converges faster for environments with dense action masking.

10. **Report on Codex reconciliation**: The two background Codex agents' findings (dataset realism and generator logic) are fully consistent with the primary analysis. No contradictions found. Both independently identified the utilization calibration bug as root cause.
